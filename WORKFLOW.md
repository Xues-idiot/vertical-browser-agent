# Spider (垂直浏览器Agent) - 自主工作流

> ⚠️ **这是作品集项目，不是demo**
> ⚠️ 每个功能都要有完整的前端界面，能实际演示
> ⚠️ 前后端同等重要，缺一不可

---

## 零、产品定位（最重要）

### Spider 是什么
**垂直场景的浏览器自动化Agent** - 不是泛用型爬虫，是专注JD筛选/简历匹配的垂直场景

### Spider 的核心竞争力
**垂直场景深度优化**：JD解析 → 简历匹配 → 智能排序 → 批量导出

vs browser-use：browser-use 是通用型，Spider 是**专注HR场景的结构化输出**
- browser-use：给你一个浏览器，你能做任何事
- Spider：给你JD和简历，直接返回排序结果

### Spider 的杀手锏
**精准匹配 + 批量处理**：
- JD结构化解析，不是关键词匹配
- 简历智能评分，可解释可追溯
- 批量处理100+简历，排序输出
- 和 HR 的日常工作无缝衔接

### 面试怎么答"和xxx有什么区别"
- 和browser-use比：Spider专注HR场景，不是通用爬虫
- 和智联/Boss直聘筛选比：AI智能匹配，不是关键词排序
- 和猎头比：Spider处理量更大，速度更快，成本更低

## 核心原则：作品集标准

> 核心理念：像 Sigma 学习，自主循环、自主进化、持续改进
> **但更重要的是：做出能拿出手给面试官看的作品**

**作品标准**：
1. 后端Agent能正常工作（调用API有返回）
2. 前端有完整的用户界面（不是空白页面）
3. 前后端能联调（不是各自独立）
4. 能拿出手给面试官看（不是半成品）

**检查清单 - 每实现一个功能必须同时完成**：
- [ ] 后端：Agent代码 + API接口
- [ ] 前端：操作界面 + 状态展示
- [ ] 联调：前后端能正常通信
- [ ] 体验：加载状态、错误处理
- [ ] 演示：功能可以实际运行展示

**必须思考的问题**（不只是完成功能）：
1. **Spider 的核心竞争力是什么？** 比 browser-use 好在哪儿？
2. **数据从哪来？** JD数据、简历数据、匹配结果...捏！
3. **为什么要做这个功能？** 是因为参考项目有这个，还是真的有需求？
4. **竞品会怎么评价？** 面试官问"和 xxx 有什么区别"怎么答？

**常见错误**：
- ❌ 只写后端，不管前端
- ❌ 前端只有一个空白页面
- ❌ 前后端不通信，各玩各的
- ❌ 功能实现了但界面无法演示
- ❌ 觉得"能跑通就行" - **作品级标准是"能拿出手"**
- ❌ 觉得"没什么可优化的" - **去用参考项目，找差距**

**数据捏造**：
- 可以用 mock 数据，只要真实感强
- JD描述、简历内容、匹配分数都可以捏
- 重点是呈现产品思路，不是真的接 API

## 九、周期性复盘审查流程（产品团队思维）

> 每轮自主循环时执行，确保代码精简、功能实用

### 循环阶段

```
┌─────────────────────────────────────────────────────────────┐
│  1. 复盘审查 (Review)    →  2. 优化代码 (Optimize)         │
│         ↓                                                       │
│  4. 添加功能 (Feature)  ←  3. 修复Bug (Fix)                  │
└─────────────────────────────────────────────────────────────┘
```

### 复盘审查清单

**组件审查**：
- [ ] 组件是否在页面中被 import 使用？
- [ ] 组件功能是否有其他组件可替代？
- [ ] 组件代码是否过于复杂（>300行）？

**功能审查**：
- [ ] 功能是否服务于核心流程（JD解析→简历筛选→报告生成）？
- [ ] 功能是否真的被用户需要？
- [ ] 功能实现是否过于复杂？

**代码质量审查**：
- [ ] 是否有未使用的 import？
- [ ] 是否有空的 useEffect 或 useCallback？
- [ ] 是否有重复的计算逻辑可以提取为 useMemo？
- [ ] 是否有可以提取为自定义 Hook 的逻辑？

### 审查结果处理

| 结果 | 处理方式 |
|------|----------|
| 未使用的组件 | 标记为待删除 |
| 可合并的组件 | 提取公共逻辑，合并组件 |
| 过长的组件 | 拆分为子组件或 Hook |
| 有 Bug 的代码 | 加入 Bug 修复队列 |
| 缺失的功能 | 加入新功能待办 |

### 执行节奏

- **每轮迭代**：完成 复盘→优化→Bug修复→功能 四个阶段
- **审查周期**：每 5 轮进行一次大规模功能复盘
- **发布节点**：每 20 轮整理一次进度记录到 PROGRESS.md

## 十、复盘审查标准（产品团队视角）

### 功能价值评估

**保留标准**（满足任一）：
- 核心流程必需（JD输入、简历列表、匹配结果、报告生成）
- 被多个组件依赖
- 用户高频使用

**可优化标准**：
- 实现过于复杂，可简化
- 与其他功能重复，可合并

**待删除标准**（满足任一）：
- 从未被使用（无 import）
- 功能鸡肋，收益小于成本
- 实现有更好的替代方案
